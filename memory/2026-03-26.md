## Session: Codex / OpenAI Integration

User wanted to switch Jarvis from Ollama to OpenAI (he has ChatGPT Plus subscription).

### Problem
- ChatGPT OAuth tokens lack `model.request` and `api.responses.write` scopes
- REST API (both Chat Completions and Responses) returns 401 "insufficient permissions"
- WebSocket to `wss://api.openai.com/v1/responses` returns 500 from Israel
- The official codex-cli WORKS because it uses internal transport that bypasses scope issues

### Solution: CodexCLIProvider
- Built `CodexCLIProvider` in `core/providers.py` that wraps `codex exec --json` as subprocess
- Uses `gpt-5.4` model (the latest available through ChatGPT subscription)
- Handles tool calls via prompt engineering (JSON extraction from response)
- Supports Hebrew perfectly

### Changes
- `core/providers.py`: Added `CodexCLIProvider` class, updated `get_provider()` factory
- `core/codex_auth.py`: Added `~/.codex/auth.json` fallback reader, JWT exp parser, updated scopes
- `.env`: Changed `JARVIS_OPENAI_MODEL` to `gpt-5.4`, provider stays `codex`
- Updated codex-cli from 0.115.0 to 0.116.0
- Re-authenticated via device code flow (`codex login --device-auth`)

### Key learnings
- ChatGPT subscription OAuth tokens only work with Codex-specific models (not gpt-4o-mini etc.)
- codex-cli uses a transport layer that doesn't enforce the same scope restrictions as REST API
- Israel region can cause issues with third-party OAuth implementations (proxy routing)
- `codex exec --json` outputs clean JSONL that's easy to parse
